Formative Activity: Faceted Data

Schmitz et al (2016) explain that the use of faceted values can enhance the security of information flow within a system by ensuring that the data concerned is subject to different behaviours according to the privileges that a user or a system channel might have (i.e. dynamically). They indicate that this is a way to build security into the design of a program, rather than implement it as an afterthought following an attack and also to not write it statically into program parameters. The authors refer to “secure multi-execution”, which works by protecting sensitive information from being written to public channels and vice versa, which effectively ensures that the processes do not interfere with each other.

Schmitz et al. (2016) explain that “Most information flow mechanisms fall into one of three categories: run-time monitors that prevent a program execution from misbehaving; static analysis techniques that analyze the whole program and reject programs that might leak sensitive information; and finally secure multi-execution, which protects sensitive information by evaluating the same program multiple times.” This is a fantastic explanation of what makes faceted information flow different.

In the context of this course, this approach seems to be ideal for enforcing role-based access control (RBAC) in a dynamic way to control what people can see without risking leaking sensitive data. It also facilitates ‘policy-agnostic programming’ which frees developers from the burden implementing complex logic in order to enforce privacy policies, and instead allows them to achieve this dynamically through applying pre-defined security ‘labels’ to sensitive data within the program itself (Mickinski et al., 2020). This is achieved through dynamically ‘labeling’ output according to continuous evaluation of values generated through program execution and is through the subsequent application of specific semantic parameters.

My further research has led me to identify that the main con of using a dynamic and faceted approach to the flow of information in a program is that it is complex and that it causes higher performance overhead than a static approach would (Mickinski et al., 2020). This is because information flow is analysed at runtime in a dynamic way. Other cons are that the ongoing faceted evaluation of information flows may also delay the discovery of possible policy violations (Mickinski et al., 2020). It is also not very commonly adopted, which means that developers in general are not familiar with the techniques involved.

Pros of a faceted approach to information flow include increased and more dynamic control over the flow of information (i.e. violations or vulnerabilities are not patched after an event has occurred) and that security requirements can be implemented in a more dynamic way, without having to statically enhance or extend roles or role-permissions for example.

References:

Micinski, K., Darais, D., and Gilray, T. (2020) “Abstracting Faceted Execution.” 2020 IEEE 33rd Computer Security Foundations Symposium (CSF), Boston, MA, USA, 2020, pp. 184-198, doi: 10.1109/CSF49147.2020.00021.

Schmitz, T., Rhodes, D., Austin, T.H., Knowles, K., Flanagan, C. (2016). Faceted Dynamic Information Flow via Control and Data Monads. In: Piessens, F., Viganò, L. (eds) Principles of Security and Trust. POST 2016. Lecture Notes in Computer Science(), vol 9635. Springer, Berlin, Heidelberg. https://doi.org/10.1007/978-3-662-49635-0_1
