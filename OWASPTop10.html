OWASP Top 10 Web Application Security Risks (2021): A03 Injection

One common coding weakness identified by OWASP is ‘Injection’ and particularly an SQL injection. To provide an explanation of how an attacker might engage in malicious injection, Williams (n.d.) writes that:

“Injection is an attacker’s attempt to send data to an application in a way that will change the meaning of commands being sent to an interpreter. For example, the most common example is SQL injection, where an attacker sends “101 OR 1=1” instead of just “101”.

In this example this is problematic, as when an attacker sends such an SQL query, they will receive information about all relevant records in the database, rather than just one. Furthermore, as many there are many interpreters (not just SQL) in a normal web environment and these have a ‘common interface’, the risk that untrusted and unsanitized data can flow through an application and through interpreters until it is eventually executed is high (Williams, n.d.).

Companies can prevent SQL injection attacks by using prepared statements (to ensure that the attacker cannot change the logic of a query), stored procedures (where the SQL code for the procedure is stored in the database and the parameters are therefore automatically executed), allow-list input validation (to ensure user input is within a specified set of allowed values), or escaping all user-supplied input (which will remove any special characters from user input which may manipulate the system) (OWASP Cheatsheets Series Team, 2021).

I think the most appropriate UML models to present the design of software would be a use case diagram and a sequence diagram. The use case diagram will show how the actors will interact with the system, meaning that we can begin to envision the right controls which will need to be in place to prevent malicious activity. The second would be the Sequence Diagram to gain a picture of the system workflow.
