Secure Software Development

Summaries and reflections on key topics, learnings, progress and assignments completed within this Module

Unit 1: OWASP Top 10 (2021) and UML Modelling for Security

This unit focussed on introducing the topic of secure software development by exploring the OWASP Top 10 Web Application Security Risks . Our primary formative task was to produce a short academic post about a particular vulnerability from this list and a flowchart and UML diagram which would explain how this vulnerability might be exploited by a hacker. Read the post here.

At the end of Unit 1, I am feeling overwhelmed with the amount of documentation and best practice principles there are to navigate in order to learn how to design secure software. However, I feel that this is a very realistic way to start, as secure software design is a vast topic that can really slow down the development process, contrary to society's general preferences of quick and resource-efficient design. It seems essential that we must learn to be more meticulous in our approach to designing software.

Unit 2: Agile Software Development and Security Standards

In Seminar 1, we discussed the agile framework and how suitable this is or how it can be adapted to include secure software design and development principles. This is not an easy task, as the iterative agile development process means that security can become a forgotten, secondary or non-continuous task (especially due to its inherent resource-intensiveness and the fact that the full project requirements are not always known in advance, or may change throughout the project). However, in an article by Sharma, A. & Bawa, R. K. (2022) we learned that the Microsoft Security Development Lifecycle Process and the OWASP CLASP Framework can be applied well to the various phases of agile software development, mainly because they do not require a sequential process like other security models.

For our formative task, we took a critical look at the ISO/IEC 27000:2018 Security Standard for Information Security Management Systems (ISMS) and produced a blog post with our thoughts on how cybersecurity threats can be prevented from within an organisation (i.e. through managing people and processes). My blog post can be read here.

At this stage, I am now feeling more familiar with vulnerabilities from a theoretical standpoint, but am looking forward now to seeing how this can work in practice and in learning some secure coding techniques in the next unit.

Unit 3: Programming Languages: History, Concepts & Design

One particularly interesting aspect of this unit was learning more about the Church-Turing Thesis. This research and theory provided the fundamental principle in comuter science by claiming that any computational problem-solving process can be simulated by a machine. If we consider the recent bounds forward in Artificial Intelligence, it is very relevant when considering the limits of what computers can and cannot do.

The primary implication is that any task can be solved algorithmically by a computer program, the only limitations being time and memory.

In terms of programming concepts, we began to look into Regex and Recursion. You can find code related to my attempt at the Towers of Hanoi problem in my GitHub repository.

Learn more

We also completed our group discussion on the OWASP Top 10 - see my Summary Post about what I found out about SQL Injection here.

At this stage in the course, I am enjoying starting to work with code again after some quite theoretical and standard-heavy input on security aspects. It has also been really interesting to consider the security of Python as a programming language, as well as other languages and to take note that it really is down to the skill of the developer in writing truly secure code. You can read my thoughts on this here.
