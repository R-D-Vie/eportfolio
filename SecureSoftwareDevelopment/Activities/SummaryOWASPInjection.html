Summary Post for Group Discussion: OWASP Top 10 (Injection Attacks)

It appears that injection errors are a consistent problem in programming, extremely widespread and that no mainstream language provides a safe way to handle them (Cifuentes & Bierman, 2019). It is then down to the skill of the developer to be aware of good programming practices to avoid SQL injection, in particular. My original post was focused on SQL Injection and I will continue to summarise what I have discovered about the topic, focussing on possible solutions.

Validation controls can be a solution to ca. 90% of injection attacks, if implemented correctly (OWASP, 2019). Benita (n.d.) explains for example that any time there is user input entered as a database query there is a risk of SQL injection vulnerability, and one way to protect against this for strings is by escaping the value by using quotation marks. The same author also notes that, especially for Python, it is commonplace today to use database adapters which usually come with tools for preventing SQL injection by using query parameters instead of plain string interpolation (Benita, n.d.). In fact, Pillai (2017) also notes that in Python, %s interpolation should be avoided in favour of template strings in order to improve the security of the language. OWASP (2019) further recommends that, to protect against (SQL) injection vulnerability, input data should at the very least be sanitized or filtered and the range or length checked and that output data should be escaped or encoded as close as possible to the interpreter.

My colleague Tomas pointed out that Object Relational Mapping can be used as a technique to conceal SQL statements within queries and that using frameworks such as this and libraries can help developers to implement validation controls and ensure that valid data is being stored in the database. He also pointed out that it is vital that developers respect the framework standards in order for this to be a valid solution. In fact, OWASP (n.d.) supports this assertion, saying that ORM generated objects can also be vulnerable to SQL Injection attacks if the methods implemented take unsanitized input parameters. I therefore wholeheartedly agree with Tomas’s suggestion that such protection methods can only be successful if developers know how to use them and respect their standards, rather than relying largely on their own experience. I would also like to briefly point out the issue of cognitive load when using external libraries and frameworks, as discussed by Cifuentes & Bierman (2019). Unless developers are familiar with these frameworks, there is a risk that this could affect their ability to implement it which in turn has an effect on cost and duration of software development projects.

In summary, it appears that the only valid way to really protect against SQL injection attacks is to use external libraries, frameworks or database adapters, but that the successful implementation of these relies heavily on the developer themselves. There is therefore a fundamental need to ensure developers with specific skills are hired or developers are thoroughly trained during a software project requirements gathering and design phase.
