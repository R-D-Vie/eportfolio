UML

UML is not a programming language per se (there is some discussion around the usefulness of using UML to inform on programming), but tools exist to turn UML diagrams into code. In this module we use UML for behavioural modelling for designing systems and this is it’s most popular application. It has been difficult to maintain a standard notation for UML and most practitioners tend to use it to define a meta-model (i.e. a diagram/ class diagram) that defines the concept.

Fowler (2013) says that the UML development process can be either waterfall or iterative. Waterfall is step by step from analysis of the problem, system design, coding and then test (done in exactly that order). Iterative is more common and means that you are going to solve each different pieces of functionality in separate iterations – you analyse, design, code and test each iteration before moving on.

Class diagrams are the most commonly used format in UML as they are an effective a way of designing and documenting an object-oriented system. A class diagram describes the types of objects and the various relationships that exist among them within a system. It also shows the properties and operations of a class and constraints which apply to the way the objects are connected.
