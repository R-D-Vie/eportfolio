Database Design

A database can be described as a colleciton of related data, designed in a certain way to suit the particular needs of an enerprise or organiation (Connolly & Begg, 2015). There is therefore inherently not 'one-fits-all' solution and a database must be designed with the particular organisation in mind and how the data should be represented and what relationships between data are necessary to highlight.

For this reason, organisations might be thought of in a similar way to classes within an object-oriented program design: each separate component or entity interacting within that system will also include its own people, activities, functions and methods.

Relational Database Design ,according to Connolly and Beg (2015), is intended to group specific attributes of data in such a way that data is not rendered redundant. This means that each attribute of data should ideally be represented only once within any database design, in order to avaoid duplications (and therefore a risk of inconsistent data deletion, adaptation or update).

Separate data in a database or system will have some sort of dependency on another piece of data - this may be functional or transitive dependency. Functional dependency represents a constraint - i.e. one will not exist or will require updating if another is changed. Transistive dependency is when data is dependent on another piece of data via an intermediary piece of data.

Normalization is a technique employed to uncover the types of dependencies (funtional or transitive) between data. It involves indentifying the 'primary keys' within a set of data, those data on which other data are functionally dependent and then taking steps to rationalise the data set to remove non-necessary (partial or transitive) data dependencies

Unnormalised Form: is when a table of data contains data duplicates or repeating groups. First Normal Form is the process of removing partial data dependencies. Second Normal Form removes transitive dependencies so that a all non-primary data are fully dependent on the primary key. Second Normal Form goes one step further to remove transitive data dependencies.

Entity Relationship Diagrams

An Entity Relationship (ER) Diagram is a way of visualising and achieving the normalisation of data within a system - it is a type of flowchart that illustrates how “entities” such as people, objects or concepts relate to each other, shown as separate tables of data. In terms of notation, it shares some similarities with UML, however it also has it's own special approach especially concerning how it identifies the 'Primary Keys' and 'Foreign Keys' (keys that are a 'primary key' in another data table) within a data set.
