Evaluation of the final project vs. the design proposal

I was part of the team ‘SecureSpace’ (Team 3) and we designed an in-mission healthcare monitoring application for astronauts on board the International Space Station (ISS) in order to support the crew in monitoring their health on a continuous basis to avoid a situation which may cause premature abandonment of a mission due to crew ill-health.

As a team, we took a very thorough approach at the design stage, which meant that by week 6 we had already got a UML class diagram and an Enhanced Entity-Relationship (EER) model for the database design in place. This was immensely helpful in distributing and starting on the coding tasks, as we roughly knew already how the different parts of the system would interact with each other. We agreed to make the system as modular as possible, and divided all of the system functionality into its own encapsulated units. We referred to this as a ‘microservice’ approach, which was criticised in the code demonstration in week 11 by the moderator as our system did not support the exchange of messages or protocols between modules. Despite the fact that our terminology might not have been entirely accurate, the intention to build each module as a ‘service’ which would link the various system functions without unnecessary overlap was in line with microservice ideology and intended to facilitate secure and contained scalability into the program.

One omission we did intentionally make (which was picked up on during the code demonstration) was our decision not to encrypt the entire database, despite that fact that this is recommended by OWASP as an essential security measure within the remit of the 2017 Top 10 Application Security Risks under the section “A3:2017 Sensitive Data Exposure”. However, as a team we decided that within the domain and the functional environment of the system we delivered, this step did not bring value. This is because the application was designed to be deployed on a standalone terminal onboard the ISS, and did not require onboard internet network connectivity. As we note in the README “If superadmin access details were leaked, there would be no way to access the database without access to the onboard terminal which is located in space, onboard the ISS.”

The only other change we made to the application design was in the function of the “Common Actions”, which we originally intended would handle all functions and methods related to user actions. However in the final implementation, we renamed this class to ‘ActionController’ in a way that it would act as an interface between the Command Line Interface and the system objects by calling those object’s methods to facilitate system actions.

Other minor changes included updating the planned role permissions to better-suit system functionality and mapping permissions to the user actions directly. We also specified the format for record download as .txt.
